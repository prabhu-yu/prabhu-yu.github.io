  1. There are many reasons on why it helps significantly if an Embedded Software Developer studies scripting languages like Python/Javascript?
    Scripting allows us to express more thoughts in less code. They manage a lot things like data typing, garbage collection, printing exact line upon runtime error etc at zero cost for us. They are mode for our (developers) productivity. In compiled language, you would modify code, build it and run it. This build state an time consuming, distracting. But in scripting language, there is not build stage, thus saving huge amount time. As said earlier, generally scripts uses dynamic typing (They are as strongly typed as C/C++). So, you are writing very less code for same feature compared to statically typed languages like C/C++/Java. I feel, type safety of compiled languages harmed us more than it helped us. To be more objective, if you write a 1 line of C code and another line of Python code, typically, you would be expressing 10 to 100 times more powerfully in python, because, to translate 1 line of python into machine code, it takes 10 to 100 time more machine codes than C. This is widely observed.

    Yes, I hear, they are not as fast as C/C++. And, don't forget, developers are also costly. Scripting languages provide function binding of C/C++. ie, we can call C++ function, in Python. Thus, the code that really needs to be optimized can be kept in C++, while rest of the code can be in Python. Today, cpu and RAM are cheap, but not the developers. We can compensate any slowness, if at all that matters by using a little faster CPU and little more CPU. That is very cheap. Give what your customers need. Majorly it is our quality product at competitive price is the need. For this, most of the time, scripting will help you (rather compiled language like C/C++/Java).

    Growth of scripting languages is phenomenal. You can see it https://pypl.github.io/PYPL.html and https://www.tiobe.com/tiobe-index/

    I faced one issue with Python/Javascript: If you write code, it is available in plaint text to every one. you can't really protected your code. This is generally not an issue in compiled languages. People say, we can send the byte code but I yet to explore this.

  6. What is a Static WebPage/Website?
    Generally, Static webPage are compared with Dynamic WebPage/Website.
    All static websites can be as dynamic as any Dynamic website, but it uses, the client site Javascript. Before we understand, what it is, let us see why we need it.
    It is almost functional equivalent of Dynamic web page, and comes with these too.
    - Extra ordinarily Simple
    - implicitly Secure
    - Highly economical (almost you need no money to setup)
    - Fast
    - Nearly resistant to attacks like DoS or DDoS
    - As dynamic as any other Dynamic webpage or website.
    - You never need a expert to setup and manage it.
    If all the pages are served statically, such webpage is called webpage. If all pages served by a website are static webpages, then whole website may be called static website
    How this matter?
    We can write all files for our website once and serve them to users as they are.
    We still need a server, but its job is just to hand over the files, as they are.
    thus, server job is similar to that of file hosting server or CDN.
    Oh, I need dynamic content in my website ...
    Yes, that is totally possible.
    How? we can use JAvascript at client side to provide dynamic feature.
    It is a very old concept but getting mainstream attention.
    Static websites are offer same functionality as that of Dynamic websites, yet, they (=static website) are far simpler, secure, safe, easy to manage, does not require very sophisticated knowledge to develop and maintain. Very first question that comes to mind is , how come static website can offer the dynamic nature of a web page. Answer is simple and well known : everything is handled at client side using JavaScript. The client script will directly talks to amy needed server instead of a dedicated webserver. Infact, in the static website, the web-server will serve the static pages only. Nothing else. No code runs on it. It is like file server. that is all. Any database, will run at a server. the client java script will communicate to it (certainly, by autheticating) Then does what ever it wants. If see properly, there is no fixed web technologies you need to follow as in Flask, Django etc. It is upto developer to do what is best.
    Reference: https://www.internetingishard.com/html-and-css
    https://www.smashingmagazine.com/2015/11/modern-static-website-generators-next-big-thing/

  9.                        
                               

                                    Modern CPUs are very complex beasts. I am trying to see, if we make them simple enough to be manufactured by                                 by a 3rd world country, yet we can depend it for almost all computings. Let us give a try.                                                                 Imagine an existing RISC cpu.                                 Remove all caches.                                 Remove all branch predictions or other speculative execution engines.                                 Remove all pre fetching.                                 Remove all kinds of complicated bus matix to CPU, just use direction connection to memory bus.                                 We may be taking the CPU back to 50 yers, but please hold.                                 Let us have 'c' logical cpu, directly connect to the off the shelf RAM                                 Idea is, since it is too cheap to implement more cpu, then to speed up single cpu to higher clocks.                                 And we have more threads in typical desktops (3000+ threads), yet, we generally have only 8 logical cpus in our desktops.                                 So, even if we increase the single cpu speed, we have 99.9% of SW sittle idle.                                 Instad of this, I want, to implement thousands of cpu, at the clock rate nearly equal to RAM speed and run thousands of threads in parallel.                                 here, instead of running a thread at very high speed, we have thousands of threads running at low speed.                                 At the end, we have nearly same performace.                                 Idea is not to increase the CPU clock, but to bring it down to the level of memory.                                 By this, we don't need to worry on idle CPU, but can have immune CPU.                                 AT the same time, if we can have large number of simple CPU connected to standard ram, we enable parallisam.                                                                 What are the cons of this.                                 Obviously die size increases becos of thousands of slow cores. but we have made enoug room by removing other components.                                 also, our clocks can be kept at constant speed, something like, run at full speed or stop them.                                 Due to reduced clock speed, our problems are less.
